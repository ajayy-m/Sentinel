# Phase 1 Run Guide

This verifies agent -> transport -> collector flow on one machine.

## 1. Install dependencies

```powershell
pip install -r requirements.txt
```

## 1.5 Single-command startup (collector + agent + UI)

Use this when you want to run the full local stack from one command:

```powershell
python sentinel.py stack --config config/config.yaml
```

VS Code one-click option:
- Run task: Sentinel: Start Full Stack

PowerShell one-click script:

```powershell
.\scripts\start_sentinel_stack.ps1
```

## 2. Start collector (terminal 1)

```powershell
python sentinel.py collector --config config/config.yaml
```

Expected behavior:
- Logs show collector listening on configured endpoint.
- JSON payloads appear once agent starts.
- Collector also answers discovery broadcasts from agents that start later.
- Collector also scans the LAN for active agents and records newly discovered nodes before their first payload arrives.

## 3. Start agent (terminal 2)

```powershell
python sentinel.py agent --config config/config.yaml
```

Expected behavior:
- Logs show periodic payload sends.
- Collector terminal prints received payloads.
- If the collector is not reachable immediately, the agent keeps discovering and reconnecting automatically.
- Agents also answer collector discovery scans so the collector can enumerate running nodes dynamically.

## 4. Optional: override config at runtime

Example endpoint override:

```powershell
$env:SENTINEL_ENDPOINT = "tcp://127.0.0.1:6001"
python sentinel.py collector --config config/config.yaml
```

Agent must use the same endpoint override to connect.

## 5. Stop processes

Press Ctrl+C in each terminal.

## 6. Dynamic fleet limits

Configure limits in config/config.yaml:

- collector.max_nodes_soft: warning threshold for unique active node IDs.
- collector.max_nodes_hard: hard threshold for unique active node IDs.
- collector.hard_limit_mode:
	- warn: allow new nodes beyond hard limit and log warnings.
	- block: reject payloads from new nodes beyond hard limit.

Example:

```yaml
collector:
	print_payload: true
	max_nodes_soft: 20
	max_nodes_hard: 50
	hard_limit_mode: "warn"
```

## 7. Storage and change detection

Collector now stores payloads and detected change events in SQLite.

Relevant config keys:

- collector.storage_enabled
- collector.sqlite_path
- collector.process_count_delta_threshold

Example:

```yaml
collector:
	storage_enabled: true
	sqlite_path: "./data/sentinel.db"
	process_count_delta_threshold: 25
```

Expected behavior during run:

- Collector logs `Change detected ...` when node fingerprint changes or process count spikes.
- SQLite DB file is created at configured path.

## 8. Query stored data (report mode)

After running collector + agent for at least a few payloads:

```powershell
python sentinel.py report --config config/config.yaml --limit 5
```

Filter by node ID:

```powershell
python sentinel.py report --config config/config.yaml --node-id local-node --limit 10
```

Compact output (recommended once dataset grows):

```powershell
python sentinel.py report --config config/config.yaml --limit 10 --summary-only
```

Compact payload previews without full metric trees:

```powershell
python sentinel.py report --config config/config.yaml --limit 10 --compact-payloads
```

Expected behavior:

- Prints JSON summary with active node count.
- Shows recent payloads and recent detected change events.
- Shows recent health summaries with status and score (including anomaly impact).
- Shows recent alerts when thresholds are breached.

## 9. Pipeline scoring thresholds

Adjust scoring thresholds in config/config.yaml under `pipeline`.

Example:

```yaml
pipeline:
	cpu_warn_percent: 80
	cpu_critical_percent: 95
	memory_warn_percent: 80
	memory_critical_percent: 95
	disk_warn_percent: 85
	disk_critical_percent: 95
	warning_score_threshold: 70
	critical_score_threshold: 40
```

## 10. Rolling trend and anomaly scoring

Trend summaries and anomaly placeholders are produced for each payload.

Config section:

```yaml
anomaly:
	window_size: 20
	min_samples: 8
	zscore_warn: 2.0
	zscore_critical: 3.0
```

For a quick anomaly test, temporarily lower thresholds:

```yaml
anomaly:
	window_size: 10
	min_samples: 3
	zscore_warn: 0.8
	zscore_critical: 1.2
```

Then run report mode and check:

- recent_trend_summaries
- recent_anomaly_scores
- recent_alerts (category anomaly)

## 11. Root-cause hint correlation (stub)

Root-cause hints are generated by correlating recent change events and anomaly scores.

Config section:

```yaml
root_cause:
	window_seconds: 60
	cooldown_seconds: 30
	min_repeated_anomalies: 2
	min_metric_instability_count: 3
```

Check in report output:

- recent_root_cause_hints
- latest.root_cause_hint (when using --summary-only)

## 12. Action recommendation stubs (non-executing)

Recommendations are derived from recent alerts and root-cause hints, but nothing is executed.

Config section:

```yaml
recommendations:
	cooldown_seconds: 30
```

Check in report output:

- recent_action_recommendations
- latest.action_recommendation (when using --summary-only)

## 13. Approval gate layer

The approval gate stores explicit action requests and approval decisions.

Config section:

```yaml
approval_gate:
	request_cooldown_seconds: 30
	auto_promote_recommendations: false
```

Report output fields:

- recent_action_requests
- recent_approval_decisions
- latest.action_request
- latest.approval_decision

Note:
- The gate is non-executing by design.
- Recommendations must be manually promoted into action requests and explicitly approved before any future executor is allowed to run.

## 14. Launch the UI dashboard

```powershell
python sentinel.py ui --config config/config.yaml
```

Tip:
- Right-click a row in Recommendations to promote it into an action request.
- Right-click a row in Requests to approve or reject it.

## 15. Scale test with synthetic nodes

Use the built-in simulator to emulate multiple nodes without deploying extra agents:

```powershell
python sentinel.py simulate --config config/config.yaml --nodes 20 --interval-seconds 1 --duration-seconds 120
```

Recommended workflow:
- Terminal 1: run collector
- Terminal 2: run simulator
- Terminal 3: run UI or report summary

For a quick sanity run:

```powershell
python sentinel.py simulate --config config/config.yaml --nodes 5 --interval-seconds 1 --duration-seconds 10
python sentinel.py report --config config/config.yaml --summary-only --limit 10
```
